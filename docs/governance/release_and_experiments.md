## Operator workflow (minimal)

1. **Experiments:** Append or update rows in `experiment_registry.json` using the `ExperimentRecord` schema (or call `upsert_experiment` from tooling). Use `query_experiments` for filters (domain, status, tags, notes substring, linked release).
2. **Release candidates:** Append `ReleaseCandidate` entries to `release_ledger.json` (or build in Python and `write_release_ledger`).
3. **Evidence bundle (config fingerprint + optional diff):** `python3 scripts/build_release_evidence_bundle.py --out evidence.json` (add `--baseline app/config/default.yaml` to include a structured diff vs that YAML merge). For shadow/live promotion, pass **`--fault-stress-run-id`** (repeatable) and **`--fault-profile-satisfied`** for each canonical profile id exercised (see `orchestration/fault_injection_profiles.py`). Attach or reference `evidence.json` from the promotion PR.
4. **Control plane:** `GET /governance/release-evidence` returns the running process bundle; `POST /governance/release-evidence/diff` with body `{"baseline_yaml": "<full default.yaml text>"}` returns a **structured report** (FB-CAN-057): `canonical_diff` (same shape as before), `semantic_analysis` (domain buckets, breaking type mismatches, operator-review flags), `replay_compatibility` hints, and `markdown_render` for PR/runbook paste. Set `"append_audit": true` to append one JSON line to `models/registry/config_diff_audit.jsonl` (local, gitignored). `GET /governance/config-diff-audit` returns the last N audit entries.
5. **Experiments (FB-CAN-027):** Use `GET /governance/experiments` with optional query filters; `POST /governance/experiments` with a full `ExperimentRecord` JSON (mutating routes require API key / session as other control-plane mutators). Linking `linked_release_candidate` to a `release_id` requires that release to exist in `models/registry/release_ledger.json` first; the ledger’s `linked_experiment_ids` is updated automatically. Status transitions follow the spec §4 table (invalid transitions return 422).
6. **Experiment completeness (FB-CAN-054):** Records in `completed`, `candidate_for_shadow`, or `candidate_for_release` must include substantive predeclared success metrics (`metrics_defined_before_run`, each token ≥2 chars), a `success_decision` narrative, and `failure_modes_observed` (minimum documented length). `upsert_experiment` rejects incomplete records. Release promotion beyond **research** with non-empty `linked_experiment_ids` fails gate `linked_experiments_complete` unless each linked id exists in `experiment_registry.json` and passes the same checks (use `scripts/validate_release_gates.py --experiment-registry …` or `POST /governance/release-objects/evaluate-gates` with optional `experiment_registry_path`).
7. **Promotion:** Run `python3 scripts/validate_release_gates.py --candidate <file.json> --target live` (or `shadow` / `simulation`). Exit code `0` = gates pass; `2` = blocked (see printed JSON for `blocked_gates`). Or `POST /governance/release-objects/evaluate-gates` with `{"candidate": {...}, "target_environment": "live"}` (optional `experiment_registry_path` for linked-experiment validation).
8. **Release objects (FB-CAN-051):** Use `GET /governance/release-objects` to read `models/registry/release_ledger.json`; `POST` upserts by `release_id`. Kinds: `config`, `logic`, `model_family`, `feature_family`, `combined`. Feature-family live promotion requires `feature_family_refs`, rollback may list `target_feature_family_refs`, and evidence must set `feature_family_replay_passed: true`.
9. **Environment stages (FB-CAN-052):** Set `environment` on the candidate to the stage **before** the gate you evaluate: `research` → promote to `simulation`; `simulation` → `shadow`; `shadow` → `live`. Skipping (e.g. `research` while targeting `live`) fails gate `environment_stage_order`. The CI fixture uses `environment: "shadow"` for `--target live`.
10. **Live–replay equivalence (FB-CAN-030):** For `kind` **logic** or **combined**, `evidence.live_replay_equivalence_passed` must be **true** for **live** promotion. Populate `live_replay_equivalence_report` with the JSON from `LiveReplayEquivalenceReport.to_dict()` when attaching harness output. CI runs `scripts/ci_live_replay_equivalence.py` (two identical replays must match decision-output fingerprints).
11. **Shadow comparison (FB-CAN-038):** For **shadow** and **live** promotion, set `evidence.shadow_comparison_passed` to **true** after running a paired replay (baseline vs candidate `logic_version` on the same dataset) and verifying divergence rates are within `apex_canonical.domains.shadow_comparison.thresholds` and `bars_compared >= probation.min_bars`. Use `python3 scripts/compare_shadow_replays.py` or `POST /governance/shadow-comparison/run` (persisted report). Attach `evidence.shadow_comparison_report` with the JSON from `ShadowComparisonReport.to_dict()` (includes `rollback_recommended`).

## Notes

- This does **not** auto-promote registered MLflow models; CI still enforces no programmatic promotion in application code (`scripts/ci_mlflow_promotion_policy.sh`).
- Canonical config versioning remains in `app/config/canonical_config.py` (`CanonicalMetadata.config_version`); release records should reference the same version strings for traceability.
