# Research Conclusions

Tabular record of every decided question. Append only — never remove rows.
Entries reference the conversation session that resolved them.

| ID | Topic | Conclusion | Rationale | Date |

| C-001 | Backtesting scope | **Removed from repo entirely.** No backtest engine, adapter, API endpoints, or MCP tools. | Out of scope until much later. Replay *invariants* (`available_time` ordering, pure builders/features) are retained for live correctness and auditability — they are not backtest machinery. | 2026-06-10 |
| C-002 | Asset class extensibility model | **Metadata-driven, additive.** Core runtime, risk gate, storage, replay, and strategy runtime never branch on `AssetClass` enum. New asset classes require: new enum variant + instrument metadata rows + new `Collector`/`Broker` implementations. Zero changes to core. | Established in architecture — mirrors how Collector and Broker traits already work. | 2026-06-10 |
| C-003 | Strategy ↔ asset class relationship | **Capability-scoped, not assigned.** Strategies declare data dependencies and primitives they require. Compatibility with an asset class is computed from instrument metadata — never assigned by the developer. | Correct extension of the metadata-driven core; eliminates a hard-coded branching problem. | 2026-06-10 |
| C-004 | Scanner asset domain | **One asset domain per scanner instance.** Mixed-domain scanners are prohibited. | Column coherence breaks with mixed domains. Terminal execution coupling demands it: what is listed must be tradeable on the same terminal type. | 2026-06-10 |
| C-005 | Discovery Scanner vs Strategy Monitor | **Two distinct modes within the same table widget component.** Discovery = user-defined filters. Strategy Monitor = strategy-driven, shows what the strategy is currently considering. Both are tabs of the same panel; neither is privileged. | A row's presence must have an unambiguous reason. Merging them into one ambiguous tool makes both worse. | 2026-06-10 |
| C-006 | Multiple scanners simultaneously | **Supported via a scanner container that holds N independent stacked instances.** Each instance owns its own filter config, column set, asset class, data subscription, and selection state. `[+]` adds a new instance; dividers are drag-to-resize. | Pro traders monitor multiple scanner criteria simultaneously (momentum, funding rate, unusual options activity, etc.). | 2026-06-10 |
| C-007 | Terminal lifecycle | Opening a terminal **increments** demand via `DemandRegistry`; closing (top-right action button) **decrements/unsubscribes**. | Makes `DemandRegistry` ref-counting visible in the UI. Shared subscriptions when the same instrument is open in scanner + terminal (ref-count 2). | 2026-06-10 |
| C-008 | Multiple terminals simultaneously | **Supported.** Multiple terminal instances open at once, each showing a different instrument. Tabs opened from watchlist rows. | Pro traders track multiple positions simultaneously. | 2026-06-10 |
| C-009 | Terminal architecture | **Slot-based layout template.** Fixed universal slots (chart, price, order entry, positions, trade history) + a variable asset-specific slot region loaded from `LayoutTemplateRegistry[AssetClass]`. | Avoids 10× code duplication and avoids forcing fundamentally different UX (e.g. options chain) into the same component shape. | 2026-06-10 |
| C-010 | Watch-table widget | **One reusable tile-based table widget** shared by Discovery Scanner, Strategy Monitor, and dashboard venue slices. Tile rows with rounded edges, no visible column dividers, asset image optional and off by default for v1 MVP. | Single reusable primitive eliminates three separate implementations. | 2026-06-10 |
| C-011 | Asset image column | **MVP v1: images excluded entirely.** Added when asset logos/images become a priority. When added: optional per asset class — ON for DEX/NFT/crypto, OFF for equity/FX/futures/bonds. | No trader identifies AAPL by a logo. An NFT is its image. Different classes have different identity conventions. | 2026-06-10 |
| C-012 | Top-level navigation sections | **Five sections:** Dashboard, Trading, Automations, Strategy Creation, Settings. Navigation via the animated button → glass pill menu from the attached Flutter spec, ported to React/Framer Motion. | Automations added for background strategy execution (separate from manual trading section). | 2026-06-10 |
| C-013 | Navigation animation | **Port the Flutter glass-pill animation spec to React/Framer Motion.** One shared `t` scalar (0=closed, 1=open). SVG `pathLength` stroke trim for button icon and perimeters. Pill reveal via top-anchored clip. 380ms, asymmetric easing (open: `easeOutCubic`, close: `easeInCubic`). Honor `prefers-reduced-motion`. | Spec provided — exact timing, easing, geometry constants, and React implementation sketch all defined. | 2026-06-10 |
| C-014 | URL routing model | **Distinct URLs per section; deep-linkable to instrument and venue.** Dashboard and Trading are separate routes. URL = workspace state. Shareable, back/forward safe. | Enables deep linking, multi-tab, external bookmark sharing. | 2026-06-10 |
| C-015 | Dashboard structure | **Three tiers:** (1) Platform-wide derived account rollup (P&L, win rate, performance graph). (2) Per-asset-class vertical slices in a horizontal infinite-panel slider. (3) Per-venue tiles within each slice (logo + key account stats). Selecting a venue slides in a transaction quick-view. | Three-layer account hierarchy: platform-wide → asset-wide → venue-wide. | 2026-06-10 |
| C-016 | Account data tiers | **Three explicit tiers:** trading-bot-wide (derived aggregate in Postgres from all venues), asset-class-wide (derived aggregate per asset class), venue-wide (source-of-truth from venue API). Platform-wide rollup is recalculated periodically, not in real time. | Clarified from conversation: "platform-wide" means the app's own Postgres account record, not a live venue feed. | 2026-06-10 |
| C-017 | Venue account adapter | **New `AccountSource` trait** alongside existing `Collector` (market data) and `Broker` (execution). Methods: `fetch_balances()`, `fetch_positions()`, `fetch_transactions()`, `health_check()`. | Account/holdings/P&L/transactions is a third capability that does not exist in the current trait model. | 2026-06-10 |
| C-018 | Venue onboarding | **`+` button per asset/venue slice → API credential prompt → dedicated venue handler runs health-check/verification before adding.** Only allowable venues (those with implemented code) are shown — no free-form URL entry. Adding new venues requires new code. | Security boundary: only known, implemented venues can be connected. Prevents arbitrary endpoint access. | 2026-06-10 |
| C-019 | API credential scope | **Read + write required.** Read-only keys are insufficient — the system executes trades on behalf of the user. Credentials stored encrypted at rest (key management TBD — see OQ-007). | Trading on behalf requires write permissions. Read-only would prevent execution. | 2026-06-10 |
| C-020 | Transaction schema | **Per-asset-class transaction tables**, not one unified table. Platform-wide and asset-wide views are assembled by the UI from multiple tables, not by a single mega-table. | Asset classes have fundamentally different transaction fields (gas for DEX, strike/expiry for options, funding for perps). A single table either has too many nullable columns or loses asset-specific data. | 2026-06-10 |
| C-021 | Live data transport (UI) | **Not HTTP/WebSocket — use NATS directly or a NATS-backed server-sent events bridge.** Goal: lowest-latency push from backend to frontend without polling. Evaluate gRPC streaming or NATS → SSE bridge as alternatives to WS. | HTTP is too slow and request-response doesn't fit the push model. The backend already uses NATS JetStream internally — the UI transport should tap into that stream as close to the source as possible. | 2026-06-10 |
| C-022 | Trading section purpose | **Manual use only.** Watchlist population (strategy-driven list filtering) is the only automation in the trading section. Order placement is always manual. | Automated order execution lives in the Automations section only. | 2026-06-10 |
| C-023 | Automations section | **Dedicated section for strategies that run order execution automatically in the background.** User can select asset classes, search instruments, assign strategies, and manage automation instances. Separate URL from Trading. | Separates manual (user-initiated trades) from automated (strategy-initiated trades). Prevents confusion about which orders are manual vs automated. | 2026-06-10 |
| C-024 | Watchlist population | **Watchlists are always backed by one or more strategies.** A strategy that is assigned/unassigned to a panel changes the panel's subscription set. Strategies that populate lists must NOT have trade execution blocks — list-only strategies only. | Answers "how do you know what to watch" — the strategy decides. Execution-capable strategies live in Automations only. | 2026-06-10 |
| C-025 | Strategy single-asset vs list/universe | **Both supported.** A strategy can target one instrument or a list/universe. Cross-sectional strategies (compare across the universe simultaneously) are architecturally distinct from fan-out (same logic applied independently to each item). Both are valid strategy types. | Resolved from conversation — the strategy builder must support composing lists, not just individual trade decisions. | 2026-06-10 |
| C-026 | Trading workspace layout | **Full-height horizontal panel layout.** Each panel (scanner list or terminal) takes full screen height top-to-bottom, and its own width. New panels open to the right. Infinite horizontal scrollable list of panels. Panels are moveable left/right. Multiple scanners and terminals can be open simultaneously. | Pro traders use horizontal layouts with multiple side-by-side panels. Width-constrained tab-switching loses context. | 2026-06-10 |
| C-027 | Multi-display support | **Same URL, multiple browser windows/displays.** Each window shows the same infinite horizontal scroll at a different position. Bottom monitor = watchlists/scanners; top monitor = terminals. | Pro traders use 2–3 monitors. The system should not require separate URLs per display. | 2026-06-10 |
| C-028 | Panel add flow | **Prompt on add: "List or Scanner?"** Empty panel added to right. User then tags a list-only strategy (no execution block) to populate it. Scanners use user-defined filter criteria. | User controls panel purpose deliberately. Prevents accidentally adding execution-capable strategies to display panels. | 2026-06-10 |
| C-029 | Scroll isolation | **Mouse scroll wheel is isolated to the panel under the cursor.** Hovering over a watchlist scrolls only that list. Hovering over a terminal scrolls only that terminal's content. No cross-panel scroll bleed. | Critical UX requirement for dense multi-panel layouts — accidental scroll between panels is disorienting. | 2026-06-10 |
| C-030 | Strategy builder scope | **Discovery + execution.** The strategy creation suite builds both: (a) list/filter strategies (scan Reddit, Twitter, websites, on-chain data, produce a ranked list), and (b) trade-decision strategies (execute when condition is met).
