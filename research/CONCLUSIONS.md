| C-001 | Backtesting scope | **Removed from repo entirely.** No backtest engine, adapter, API endpoints, or MCP tools. | Out of scope until much later. Replay *invariants* (`available_time` ordering, pure builders/features) are retained for live correctness and auditability — they are not backtest machinery. | 2026-06-10 |
| C-002 | Asset class extensibility model | **Metadata-driven, additive.** Core runtime, risk gate, storage, replay, and strategy runtime never branch on `AssetClass` enum. | C-003 | Strategy ↔ asset class relationship | **Capability-scoped, not assigned.** Strategies declare data dependencies and primitives they require. | C-004 | Scanner asset domain | **One asset domain per scanner instance.** Mixed-domain scanners are prohibited. | Column coherence breaks with mixed domains. Terminal execution coupling demands it: what is listed must be tradeable on the same terminal type. | 2026-06-10 |
| C-005 | Discovery Scanner vs Strategy Monitor | **Two distinct modes within the same table widget component.** Discovery = user-defined filters. | C-006 | Multiple scanners simultaneously | **Supported via a scanner container that holds N independent stacked instances.** Each instance owns its own filter config, column set, asset class, data subscription, and selection state. `[+]` adds a new instance; dividers are drag-to-resize. | Pro traders monitor multiple scanner criteria simultaneously (momentum, funding rate, unusual options activity, etc.). | 2026-06-10 |

| C-018 | Venue onboarding | **`+` button per asset/venue slice → API credential prompt → dedicated venue handler runs health-check/verification before adding.** Only allowable venues (those with implemented code) are shown — no free-form URL entry. | C-020 | Transaction schema | **Per-asset-class transaction tables**, not one unified table. | C-021 | Live data transport (UI) | **Not HTTP/WebSocket — use NATS directly or a NATS-backed server-sent events bridge.** Goal: lowest-latency push from backend to frontend without polling. | C-022 | Trading section purpose | **Manual use only.** Watchlist population (strategy-driven list filtering) is the only automation in the trading section. Order placement is always manual. | Automated order execution lives in the Automations section only. | 2026-06-10 |
| C-023 | Automations section | **Dedicated section for strategies that run order execution automatically in the background.** User can select asset classes, search instruments, assign strategies, and manage automation instances. | C-024 | Watchlist population | **Watchlists are always backed by one or more strategies.** A strategy that is assigned/unassigned to a panel changes the panel's subscription set.
